# NPU ISA & Architecture — Design Q&A

Answers derived from NPUArchitectureV2 (docx + md) and ISA coverage analysis.
ISA version: v2.0 · Target: KR260 XCK26 · Date: 2026-05-18

---

## 1. How do instructions work when data needs to flow from unit to unit?

Instructions do not carry or route data. Data paths are **hardwired in RTL**; instructions tell each unit when to go.

```
DDR4
 ↓ DMA_LOAD        → Act Bank A/B    (SRAM)
 ↓ WT_LOAD         → Weight Bank A/B (SRAM)
                         ↓
                       MATMUL reads both banks → INT32 output
                         ↓
                       PSB (16×16 INT32 register file)
                         ↓  PSB_FLUSH
                       Requant pipeline (hardwired feed-forward)
                         ↓  REQUANT
                       VPU lane inputs (hardwired)
                         ↓  SIMD_ACT / ELEW_ADD / etc.
                       Output Bank (SRAM)
                         ↓ DMA_STORE
                       DDR4
```

Every arrow is a fixed RTL wire or shared SRAM port. `FENCE` enforces ordering at layer boundaries. `PSB_FLUSH` is the one explicit handoff signal crossing a control boundary.

**Mental model: instructions = triggers, SRAM banks = staging, FENCE = ordering, PSB_FLUSH = the one explicit handshake.**

---

## 2. VPU instructions — do they route data to the VPU?

No. The VPU is always in the data path. Instructions control which VPU stage is active.

```
Requant output (INT8)
    ↓
① Act LUT   ← SIMD_ACT fires it / LUT_BYPASS skips it
    ↓
② Eltwise   ← ELEW_ADD or ELEW_MUL fires it / passthrough otherwise
    ↓
③ HREDUCE   ← HREDUCE fires it / passthrough otherwise
    ↓
Output Bank
```

`LUT_BYPASS` is literally a mux select bit — no data moves. Unit-specific instructions = **mode/gate control**, not data routing.

## 3. What about operations that bypass the SA entirely (MAXPOOL, UPSAMPLE, CONCAT)?

Some YOLOv8n layers skip the SA completely:

| Layer | Operation | Units used |
|---|---|---|
| 9 | SPPF MaxPool | VPU only |
| 10, 13 | Upsample 2× | DMA only |
| 11, 14, 17, 20 | Concat | DMA only |

This requires a **VPU input mux** — the VPU cannot always receive data from Requant output. `CONFIG` sets a `pipeline_mode` shadow register before the VPU instruction fires:

| Mode | VPU lane source | SA/PSB/Requant active? |
|---|---|---|
| `CONV` | Requant output | Yes |
| `POOL` | Act Bank direct | No |
| `ELTWISE` | Requant + Residual Bank | No |
| `HREDUCE` | Output Bank / prior VPU output | No |

For bypass paths, use `FENCE` at the sequencer level instead of per-instruction dep tokens:

```
CONFIG    pipeline_mode=POOL
DMA_LOAD  ...  dep_push_next=0
FENCE     wait=[DMA_done]       // sequencer barrier replaces token
MAXPOOL   dep_pop_next=0        // don't wait on Requant RAW FIFO
```

## 4. Section 3.2 — Instruction Word Encoding

128-bit fixed-width. All fields are set by the ARM driver at compile time, not generated by hardware at runtime.

```
[127:120]  opcode          — 8-bit instruction type
[119:116]  unit_id         — routes to one of 5 per-unit FIFOs
[115]      dep_push_next   — push done-token into downstream RAW FIFO on completion
[114]      dep_push_prev   — push done-token into upstream WAR FIFO on completion
[113]      dep_pop_next    — block until token arrives from upstream RAW FIFO
[112]      dep_pop_prev    — block until token arrives from downstream WAR FIFO
[111:0]    payload         — 112-bit instruction-specific data
```

**Underlying architecture: dependency token system (modeled on Apache TVM VTA).**

Between every producer/consumer pair there are two depth-4 token FIFOs:
- **RAW FIFO** — Read After Write: downstream blocks until upstream finishes writing
- **WAR FIFO** — Write After Read: upstream blocks until downstream finishes reading

Tokens carry no data — just a 1-bit "go" signal. Think of them as semaphore credits.

There are 5 adjacent pairs × 2 FIFOs = **10 token FIFOs total**.

## 5. Can multiple units run at the same time?

Yes — that is the whole point of the five independent per-unit FIFOs.

The sequencer dispatches to all five FIFOs without waiting for any to finish. Units execute concurrently as long as dep token handshakes are satisfied.

From section 6.1 of the docx:

```
T=0  DMA: WT_LOAD+DMA_LOAD L0   SA: idle           Requant+VPU: idle
T=1  DMA: WT_LOAD+DMA_LOAD L1   SA: MATMUL L0      Requant+VPU: idle
T=2  DMA: WT_LOAD+DMA_LOAD L2   SA: MATMUL L1      Requant+VPU: REQUANT+SiLU L0
T=3  DMA: DMA_STORE L0          SA: MATMUL L2      Requant+VPU: REQUANT+SiLU L1
```

At T=2, all three hardware groups run simultaneously on different layers' data.

## 6. Are the dep flags set by instructions or passed around by units?

**Set by the ARM driver** when writing the instruction stream — completely static. The hardware units just obey whatever bits are in their instruction.

```
ARM driver (offline) → writes instruction words with dep bits already set
        ↓
DDR4 instruction buffer
        ↓
Sequencer fetches + dispatches to per-unit FIFOs
        ↓
Each unit executes: checks token FIFOs, pushes/pops per its dep bits
```

The token FIFOs are the runtime mechanism, but the policy is decided offline.

---

## 7. Choosing between two VPU inputs — how do the tokens handle this?

Two separate problems:

**Mux select (which data):** `CONFIG` sets `pipeline_mode` shadow register → hardware mux reads it.

**Token path (ordering):** For non-standard paths (bypassing Requant), dep_pop/push bits are set to 0 on those instructions, and `FENCE` at the sequencer level handles ordering instead.

- **Normal linear path** → use dep bits, fine-grained per-instruction handshake
- **Bypass path** → dep bits zeroed on bypassed units, FENCE is the escape hatch

---

## 8. One REQUANT or three?

**Recommendation: keep 1 REQUANT opcode, encode 2-bit `mode` in payload bits [111:110].**

The RequantPipeline RTL already implements three modes with a front-end input mux:

| mode | Input | When in YOLOv8n |
|---|---|---|
| `2'b01` FROM_PSB | INT32 from PSB accumulator | Every Conv+SiLU layer |
| `2'b00` FROM_SRAM | INT8 rescale to new scale | Mismatched-scale branches |
| `2'b10` BINARY_ADD | Two INT8 inputs independently scaled then summed | ELEW_ADD with different quant scales |

All three use the same `clip(round((acc × M) >> S), −128, 127)` pipeline. Only the input mux differs. Three opcodes would dispatch to identical hardware — adds ISA complexity for zero gain.

## 9. Instruction FIFOs and memory readiness

**One FIFO per functional unit** (5 total: SEQ, DMA, SA, PSB/REQ, VPU). A single shared FIFO forces strict sequential execution — no concurrency.

```
Sequencer reads DDR4 → decodes unit_id → routes to correct FIFO
     ↓       ↓       ↓       ↓       ↓
    DMA      SA     PSB     REQ     VPU
    FIFO    FIFO   FIFO    FIFO    FIFO
     ↓       ↓       ↓       ↓       ↓
  unit FSM  unit FSM  ...  (each pulls independently)
```

**Memory readiness — two levels:**

1. **AXI level (inside DMA):** AXI4 `RVALID` + `RREADY` handshake confirms each burst beat has landed in SRAM.
2. **Token level (between units):** DMA pushes RAW token after burst completes → SA (or VPU) was blocked on `dep_pop_next` → token arrives → unit unblocks → reads SRAM.

The SA never knows about AXI. The token is its only signal.

## 10. Sequencer vs per-unit control units

| Comp Arch analog | NPU equivalent |
|---|---|
| PC + Fetch + Decode | Sequencer — reads 128-bit instructions from DDR4, decodes opcode, routes payloads to unit FIFOs |
| Execution unit FSM | Per-unit control — runs the actual hardware operation |

Each unit needs a small local FSM:
```
IDLE → pop dep token → wait → EXECUTE → push done token → IDLE
```

`ControlUnit.sv` should be the sequencer. Each FU module (`SA_top`, `vpu`, `RequantPipeline`) owns its FSM internally. No god-controller — dep token FIFOs replace centralized arbitration.

## 11. Reading instructions faster than data arrives

Yes — the sequencer fetches 128-bit words in 1–2 cycles; DDR4 AXI data has tens to hundreds of cycles of latency.

**The per-unit FIFOs absorb this mismatch.** The sequencer fills FIFOs ahead of time; units block on dep tokens until data is ready.

**FIFO overflow protection — two options:**

1. **Backpressure (recommended):** Each FIFO's `prog_full` signal stalls the sequencer for that unit only. Other unit FIFOs can still accept instructions.
2. **Deep FIFOs:** Size to hold max instruction burst per layer (e.g. depth-32 handles 8 K-tile loops).

Recommended FIFO depths:

| FIFO | Depth |
|---|---|
| DMA instruction FIFO | 16 |
| SA instruction FIFO | 32 (MATMUL repeats N times per layer) |
| PSB instruction FIFO | 32 (PSB_ACC repeats with MATMUL) |
| REQ instruction FIFO | 8 |
| VPU instruction FIFO | 16 |
| Dep token FIFOs | 4 (per architecture spec) |

## 12. Do instructions come from the DMA?

No. Instructions and data are completely separate flows.

```
DDR4
 ├── Instruction buffer (0x2000_0000)  → Sequencer reads via its own AXI4 master
 └── Data (weights, activations)       → DMA reads via AXI HP0 / writes via HP1
```

The DMA only moves data. The sequencer has its own fetch path.

---

## 13. AXI-Lite signals and sequencer outputs

AXI-Lite is the **slave interface** the ARM uses to write CSR registers into the sequencer. The sequencer fetches instructions from DDR4 using a separate **AXI4 master** (AXI-Lite cannot burst).

**AXI-Lite slave (ARM → sequencer CSRs):**

| Channel | Key signals | Direction |
|---|---|---|
| AW | `AWADDR`, `AWVALID`, `AWREADY` | ARM→SEQ |
| W | `WDATA[31:0]`, `WVALID`, `WREADY` | ARM→SEQ |
| B | `BRESP`, `BVALID`, `BREADY` | SEQ→ARM |

**AXI4 master (sequencer → DDR4, 4 beats per 128-bit instruction):**

| Channel | Key signals | Who drives |
|---|---|---|
| AR | `ARADDR`, `ARVALID`, `ARREADY` | SEQ |
| R | `RDATA[31:0]`, `RVALID`, `RREADY`, `RLAST`, `RRESP` | DDR4→SEQ |

**Sequencer outputs after decoding a 128-bit instruction:**

```systemverilog
logic [111:0]  fifo_payload;    // routed to target unit FIFO
logic [3:0]    fifo_dep_flags;  // push/pop token bits
logic          fifo_push;       // push strobe to target FIFO
logic          fifo_full;       // backpressure from target FIFO
logic [5:0]    unit_done;       // one bit per unit, for FENCE resolution
```

## 14. Clock domains

Three clock domains on this design:

```
100 MHz   AXI-Lite CSR  ──CDC──>  300 MHz NPU core
                                        ↕
                         200–300 MHz AXI HP0/HP1 (DMA)
```

| Clock | Used for |
|---|---|
| 300 MHz | NPU PL fabric: SA, PSB, Requant, VPU, sequencer datapath |
| 100 MHz | AXI-Lite CSR interface: ARM ↔ sequencer registers |
| 200–300 MHz | AXI HP0/HP1: DMA engine ↔ DDR4 controller |

**Use Xilinx SmartConnect (PG247) to handle CDC automatically.** Hand-writing synchronizers for every CSR register is error-prone. SmartConnect + Clocking Wizard (PG065) handles it.

## 15. Vivado FIFO IP vs hand-written FIFOs

**Use Xilinx FIFO Generator IP (PG057) for all instruction FIFOs and skew FIFOs.**

Reasons: pre-characterized for XCK26 timing closure at 300 MHz; built-in independent-clock CDC mode; `prog_full` backpressure flag included.

**Exception — dep token FIFOs (depth 4, 1-bit):** too small to justify IP overhead. Hand-write as a shift register with counter:

```systemverilog
logic [3:0] tokens;
logic [2:0] count;
assign push_ok = (count < 4);
assign pop_ok  = (count > 0);
```

| FIFO | Recommendation |
|---|---|
| Instruction FIFOs (DMA, SA, PSB, REQ, VPU) | Xilinx FIFO Generator PG057 |
| Skew FIFOs for SA column alignment (depth 0–15) | Xilinx FIFO Generator PG057 |
| Dep token FIFOs (depth 4, 1-bit) | Hand-write |

## 16. Multiple AXI masters

Yes — Xilinx SmartConnect (PG247) arbitrates multiple masters to one DDR4 controller.

| Master | Port | What it moves |
|---|---|---|
| Sequencer fetch | Shared HP0 (low priority) | 128-bit instruction words |
| DMA read | HP0 (high priority) | Weights + activations from DDR4 |
| DMA write | HP1 | Output activations to DDR4 |
| ARM PS | PS native path | OS, driver, model weights (initial load) |

DMA is the sole data master. It uses an internal **bank select** signal in each instruction payload to route burst data to the correct SRAM bank after transfer.

---

## 17. DMA — custom RTL or Xilinx IP?

**Both.** From docx section 1.4: `2D Strided DMA — Custom RTL`.

| Component | Source |
|---|---|
| 2D strided address generator (`addr = base + h*row_stride + w*ch_count + c`) | You write it |
| Zero-padding insertion FSM | You write it |
| UPSAMPLE / CONCAT control logic | You write it |
| Ping-pong bank select | You write it |
| AXI4 burst protocol (ARVALID/ARREADY/RDATA etc.) | Xilinx AXI DMA IP (PG021) |

Your custom RTL generates descriptors (source address, length, destination bank). The Xilinx AXI DMA IP executes the actual wire-level AXI bursts. Do not hand-write AXI4 burst logic at 300 MHz from scratch.

```
Your RTL                          Xilinx IP
┌─────────────────────┐          ┌──────────────┐
│ descriptor gen      │─desc────>│ AXI DMA IP   │──HP0──> DDR4
│ addr computation    │          │  (PG021)     │
│ zero-pad FSM        │          └──────────────┘
│ bank select         │                ↓ burst data
│ UPSAMPLE / CONCAT   │           SRAM banks
└─────────────────────┘
```

## 18. Instruction fetch vs DMA — serial or parallel?

**Instruction fetch is separate from DMA.** The DMA never touches the instruction buffer.

Within a single AXI master, transactions are serial. Across masters they are parallel:

```
Cycle:     0        1        2        3        4

SEQ fetch  [instr0][instr1][instr2][instr3]...   sequential, own port

DMA HP0    [WT_LOAD burst....................]   concurrent with fetch

DMA HP1              [DMA_STORE burst.......]    concurrent with both
```

Cross-unit pipeline concurrency (from docx section 6.1):

```
T=0  DMA: loads L1 data      SA: computes L0      VPU: post-processes L(n-1)
T=1  DMA: loads L2 data      SA: computes L1      VPU: post-processes L0
T=2  DMA: stores L0 output   SA: computes L2      VPU: post-processes L1
```

SmartConnect arbitrates HP0 contention between sequencer fetch and DMA read — sequencer gets lower priority since it is not on the critical compute path.

## 19. PyTorch op → instruction sequence mapping

Every YOLOv8n PyTorch op compiles to a fixed instruction stream written by the ARM driver:

| PyTorch op | Instruction sequence |
|---|---|
| `Conv2d + SiLU` | `COEFF_LOAD, LUT_LOAD, WT_LOAD, DMA_LOAD, FENCE, MATMUL×N, PSB_ACC×N, PSB_FLUSH, FENCE, REQUANT, SIMD_ACT, DMA_STORE` |
| `Conv2d` (linear, no activation) | Same but `LUT_BYPASS` replaces `SIMD_ACT` |
| `C2f residual add` | `ELEW_ADD` appended after `SIMD_ACT` |
| `MaxPool 5×5` (SPPF) | `CONFIG mode=POOL, DMA_LOAD, FENCE, MAXPOOL×3, DMA_STORE` |
| `Upsample 2×` (FPN) | `UPSAMPLE` (DMA handles entirely) |
| `Concat` (FPN/PAN) | `CONCAT` (DMA handles entirely) |
| `DFL box decode` | `HREDUCE` (once per frame after layer 21) |

The driver iterates the YOLOv8n compute graph layer by layer, emits the corresponding stream into the DDR4 instruction buffer (0x2000_0000), writes `INSTR_COUNT` and `INSTR_BASE` CSRs, then asserts the sequencer kick bit.

## 20. Should SRAM banks be distributed inside each FU module or live in a centralized Hub at the top level?

**Centralized Hub at the top level.** Rule: if more than one unit touches a bank, it must be in the Hub — otherwise DMA would need cross-module write ports into each FU.

| Bank | Writers | Readers | Location |
|---|---|---|---|
| Act Bank A/B | DMA | SA, VPU (POOL mode) | Hub |
| Weight Bank A/B | DMA | SA | Hub |
| Residual Bank | DMA | VPU (ELEW_ADD) | Hub |
| Output Bank | VPU | DMA (DMA_STORE), VPU (HREDUCE) | Hub |
| PSB register file | SA (PSB_ACC) | Requant (PSB_FLUSH) | Inside PSB/SA — DMA never touches it |
| Requant coeff BRAM | DMA (COEFF_LOAD) | Requant only | Inside Requant module — DMA gets one write port |
| Act LUT BRAM | DMA (LUT_LOAD) | VPU only | Inside VPU module — DMA gets one write port |

On FPGA, "near the FU" is a floorplan concern (Vivado Pblocks), not an RTL concern. Write logically centralized; constrain BRAM placement physically. `[Source: UG949 §"Floorplanning"]`

## 21. What AXI4 master signals does the Sequencer need for instruction fetch?

Instruction fetch is **read-only** — only AR and R channels. No write channels.

```systemverilog
// Sequencer drives — AR channel
output logic [43:0] m_axi_araddr,   // 44-bit physical addr (Zynq UltraScale+)
output logic        m_axi_arvalid,
output logic [7:0]  m_axi_arlen,    // 8'd3 = 4 beats per 128-bit instruction
output logic [2:0]  m_axi_arsize,   // 3'b010 = 32-bit beats
output logic [1:0]  m_axi_arburst,  // 2'b01 = INCR
input  logic        m_axi_arready,

// Sequencer receives — R channel
input  logic [31:0] m_axi_rdata,
input  logic        m_axi_rvalid,
input  logic        m_axi_rlast,
input  logic [1:0]  m_axi_rresp,
output logic        m_axi_rready,
```

Internally: shift four 32-bit beats into a 128-bit register; decode fires on `rlast`. ARLEN = 3 (4 beats; AXI counts from 0). Other sequencer inputs: `csr_instr_base`, `csr_instr_count`, `csr_kick` (from AXI-Lite CSR, CDC'd), `fifo_full[4:0]` (backpressure from 5 unit FIFOs), `unit_done[4:0]` (for FENCE resolution). `[Source: IHI0022H AXI4 spec §A3]`

## 22. Does the Sequencer need an input FIFO if it runs at 300 MHz while AXI-Lite is 100 MHz?

**No.** The ARM does not stream instructions into the sequencer. It writes three CSR registers once per layer dispatch — `instr_base`, `instr_count`, `kick`. SmartConnect or a 2-FF synchronizer handles that single CDC crossing with no FIFO.

After the kick, the sequencer's own AXI4 master autonomously bursts instructions from DDR4 at 300 MHz. The sequencer is the *initiator*, not the receiver of a stream. The FIFOs in this design are all *outputs* of the sequencer — it fills the five per-unit instruction FIFOs; it drains no input FIFO.

```
ARM (100 MHz): writes {base, count, kick} once per layer
        ↓  2-FF sync / SmartConnect CDC
Sequencer (300 MHz): autonomous AXI4 burst → decode → push to unit FIFOs
```

`[Source: ISA_Architecture_QA.md §14; PG247 SmartConnect]`

## 23. Token flow — which units send and receive tokens? Does SRAM participate?

SRAM is passive — it never sends or receives tokens. Tokens flow only between the FSMs of adjacent pipeline units. The five RAW token pairs (each also has a reverse WAR token):

```
DMA ──RAW──> SA        DMA signals: Act/Weight SRAM write is done, safe to read
SA  ──RAW──> PSB       SA signals: MATMUL output is latched, PSB_ACC may fire
PSB ──RAW──> Requant   PSB_FLUSH is the explicit one-crossing handoff
Req ──RAW──> VPU       Requant signals: INT8 lane output is valid
VPU ──RAW──> DMA       VPU signals: Output Bank write is done, DMA_STORE may fire
```

A RAW token from DMA to SA means "the BRAM write address has retired and read data will be valid." The BRAM has no awareness of tokens — the token simply controls when the SA FSM issues its read address. `[Source: ISA_Architecture_QA.md §4, §9; Apache TVM VTA dependency FIFO design]`

## 24. What fields from a 128-bit instruction are pushed into the unit FIFOs?

The sequencer consumes `unit_id[119:116]` for routing, then pushes the remaining **116 bits** into the selected unit's FIFO:

```
[127:120]  opcode     ──┐
[119:116]  unit_id    ──┴── sequencer consumes for routing; never enters a FIFO
[115:112]  dep_flags  ──┐
[111:0]    payload    ──┴── 116 bits pushed into the target unit FIFO
```

FENCE and CONFIG (`unit_id = Sequencer`) are never routed to a unit FIFO — the sequencer handles them internally. CONFIG loads shadow registers; FENCE stalls dispatch until the `unit_done` bitmask in the payload clears.

## 25. Does each functional unit need its own decoder?

Each unit needs only a **small local case statement** on the opcode byte it receives from its own FIFO. Since opcodes are partitioned by unit, each case has very few arms:

| Unit FIFO | Opcodes received | Case arms |
|---|---|---|
| DMA | 0x10–0x15 | 6 |
| SA | 0x20 | 1 |
| PSB | 0x21–0x22 | 2 |
| Requant | 0x30 | 1 |
| VPU | 0x31–0x38 | 8 |

No unit ever sees another unit's opcodes — the routing guarantee means the DMA FSM never needs to know what `SIMD_ACT` is. The sequencer does routing-level decode only: read `unit_id` → pick FIFO; detect FENCE/CONFIG to handle internally. No central god-decoder at any level. This is the VTA pattern: distributed per-unit instruction interpreters. `[Source: apache/tvm-vta sequencer design; ISA_Architecture_QA.md §10]`
